# IronLint 0.1c — Claude Code Adapter Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship the Claude Code adapter — the final piece of the 0.1 release. After this, a user can `/plugin install` IronLint in Claude Code and get bully's UX: PostToolUse hook blocks bad edits, Stop hook checks session-aware rules, `/ironlint-init` and `/ironlint-author` and `/ironlint-review` skills appear.

**Architecture:** A new `adapters/claude-code/` directory contains `plugin.json`, `hooks/hook.sh`, `hooks/hooks.json`, and `skills/<name>/SKILL.md` files. The hook script is a thin bash wrapper that translates Claude Code event payloads to `ironlint check` invocations and maps verdict JSON to Claude Code's expected exit codes. The adapter is *not* Rust code — it's plugin metadata + a shell script + markdown skill files. The `ironlint` binary itself doesn't change much; the adapter just consumes its existing CLI.

**One small `ironlint` addition:** a `ironlint session record --file X --diff Y` subcommand so the PostToolUse hook can append an edit record to `.ironlint/session.json`. The accumulated state is then evaluated by the Stop hook via `ironlint check --session`.

**Tech Stack:** Bash (POSIX-ish; `set -euo pipefail`), `jq` (for parsing Claude Code event JSON), markdown (skills), JSON (plugin manifest + hooks config). No new Rust workspace dependencies.

use anyhow::Result;
use ironlint_core::session_state::{EditRecord, SessionState};
use std::path::Path;

pub fn record(dir: &Path, file: &Path, diff: &str, session_id: Option<String>) -> Result<i32> {
    let state_path = dir.join(".ironlint/session.json");
    let mut state = match SessionState::load(&state_path) {
        Ok(s) => s,
        Err(_) => {
            // File missing or unreadable — start a fresh session.
            let id = session_id.unwrap_or_else(|| format!("session-{}", chrono::Utc::now().timestamp()));
            SessionState::new(id)
        }
    };
    state.append(EditRecord {
        file: file.display().to_string(),
        diff: diff.to_string(),
        timestamp: chrono::Utc::now().to_rfc3339(),
    });
    state.save(&state_path)?;
    Ok(0)
}
```

Add `chrono.workspace = true` to `crates/ironlint-cli/Cargo.toml` `[dependencies]` if not already present.

- [ ] **Step 3.2: Create `adapters/claude-code/hooks/hook.sh`**

```bash
#!/usr/bin/env bash
set -euo pipefail

# Claude Code adapter for ironlint.
#
# Routes lifecycle events to `ironlint` invocations:
#   - post-tool-use: run `ironlint session record` to accumulate the edit,
#                    then `ironlint check --file <changed-file>` to gate the edit.
#                    Exit 2 on block, 0 otherwise.
#   - stop:          run `ironlint check --session` to evaluate session rules.
#   - session-start: clear stale `.ironlint/session.json` from a previous run.
#
# Event JSON arrives on stdin. We pipe through jq to extract paths.

MODE="${1:-post-tool-use}"

# Default project root is the CWD where Claude Code is running.
PROJECT_ROOT="$(pwd)"
CONFIG="${PROJECT_ROOT}/.ironlint.yml"

# Skip silently if ironlint isn't configured for this project.
if [[ ! -f "${CONFIG}" ]]; then
  exit 0
fi

case "${MODE}" in
  session-start)
    # Clear any stale session.json from a prior aborted session.
    rm -f "${PROJECT_ROOT}/.ironlint/session.json" 2>/dev/null || true
    exit 0
    ;;

  stop)
    # Evaluate session rules over the accumulated changeset.
    # If session.json doesn't exist (no edits this session), exit 0 without running.
    if [[ ! -f "${PROJECT_ROOT}/.ironlint/session.json" ]]; then
      exit 0
    fi
    if ! ironlint check --session --config "${CONFIG}" --format json > /tmp/ironlint-session-verdict.json; then
      cat /tmp/ironlint-session-verdict.json >&2
      exit 2
    fi
    # session.json was cleared by ironlint check --session as a side effect.
    exit 0
    ;;

  post-tool-use|*)
    # Parse the event JSON for the changed file.
    EVENT=$(cat)
    FILE=$(echo "${EVENT}" | jq -r '.tool_input.file_path // .tool_input.path // empty')
    if [[ -z "${FILE}" ]]; then
      # No file in event payload — nothing to check.
      exit 0
    fi

    # Build a synthetic unified diff for session recording.
    # (Claude Code's Edit/Write events don't include a diff; we fake one
    #  from the new_string + old_string pair when available.)
    OLD=$(echo "${EVENT}" | jq -r '.tool_input.old_string // ""')
    NEW=$(echo "${EVENT}" | jq -r '.tool_input.new_string // .tool_input.content // ""')
    DIFF="--- a/${FILE}
+++ b/${FILE}
@@ -1 +1 @@
-${OLD}
+${NEW}"

    # 1. Record the edit into session state (non-blocking).
    ironlint session record --dir "${PROJECT_ROOT}" --file "${FILE}" --diff "${DIFF}" >/dev/null 2>&1 || true

    # 2. Gate the edit by running checks.
    if ! ironlint check --file "${FILE}" --config "${CONFIG}" --format json > /tmp/ironlint-verdict.json; then
      cat /tmp/ironlint-verdict.json >&2
      exit 2
    fi
    exit 0
    ;;
esac
```

# IronLint Init

Generate a baseline `.ironlint.yml` by detecting the stack, wiring installed linters as passthrough rules, and routing project-specific rules to the right engine.

This skill is user-driven. Do not silently install tools or migrate rules. Every step below is a proposal the user accepts or declines.

## Step 1: Run `ironlint init`

The `ironlint init` command detects the stack from manifest files:

| Manifest | Stack |
|---|---|
| `Cargo.toml` | Rust |
| `package.json` | Node |
| `pyproject.toml` or `setup.py` | Python |
| (none) | Generic |

```bash

```

This creates `.ironlint.yml` with one or two starter rules for the detected stack. Review the output.

## Step 2: Add stack-specific linter passthroughs

For each linter the user has installed (ruff, biome, eslint, tsc, phpstan, clippy, …), propose a passthrough rule:

```yaml
ruff-check:

  engine: script
  scope: ["**/*.py"]
  severity: error
  script: "ruff check --quiet {file}"
```

Test each candidate by running the script against a sample file. Only add rules that succeed without spurious errors.

## Step 3: Trust the config

After review, run:

```bash
ironlint trust
```

This writes a sha256 fingerprint of the canonicalized config into `.ironlint.yml`. Future runs verify the fingerprint and refuse to execute rules from an untrusted config.

Edit any in-scope file. The PostToolUse hook should run ironlint and either pass (clean) or block (with a violation message).

# IronLint Author

Interactive authoring for `.ironlint.yml`. Every proposed rule is tested against a fixture before being written.

If no `.ironlint.yml` exists, stop and tell the user to run `/ironlint-init` first.

## Triggers

The user wants to:
- Add a new rule ("ban X", "warn on Y").
- Modify an existing rule (tighten, change scope, change severity).
- Remove a rule ("drop X").
- Convert a rule between engines (script ↔ ast ↔ semantic).

## Engine routing

Pick the engine based on what the rule needs to detect:

| Rule shape | Engine |
|---|---|
| "Run this linter command" | `script` |
| "Match this AST shape exactly" (e.g., `as any`, `eval(...)`) | `ast` |
| "Code violates this plain-English policy" | `semantic` |
| "These changes need test changes in the same session" | `session` |

## Process

1. Read `.ironlint.yml` to see existing rules.
2. Draft the new rule with appropriate engine + scope + severity.
3. Build a fixture file that exercises the rule (a clean version + a dirty version).
4. Run ironlint against the fixture:
   ```bash
   ironlint check --file /tmp/dirty.<ext>
   ```
5. Verify the rule fires on dirty input and passes on clean input.
6. If the test passes, write the rule into `.ironlint.yml`.
7. Run `ironlint trust` to update the fingerprint.

## Required fields by engine

- `script`: `description`, `engine: script`, `scope`, `severity`, `script` (with `{file}` substitution).
- `ast`: `description`, `engine: ast`, `scope`, `severity`, `pattern`, `language`.
- `semantic`: `description` (= the policy in plain English), `engine: semantic`, `scope`, `severity`. Requires `llm:` block in config.
- `session`: same as semantic, plus `context: repo` is recommended.

## Capabilities (script + ast)

By default, all script rules run with `network: false, writes: none`. If a rule needs to write within the project, add:

```yaml
capabilities:
  network: false
  writes: cwd-only
```

Don't grant `network: true` or `writes: unrestricted` unless the user explicitly asks.

## Test before write

Always test the rule against a fixture BEFORE writing to `.ironlint.yml`. A rule that doesn't fire on dirty input is worse than no rule — it gives false confidence.

# IronLint Review

Audit the rule set against telemetry. Surface candidates for removal, downgrade, or scope adjustment.

## Source of truth

`.ironlint/log.jsonl` — one record per `ironlint check` invocation. Schema:

```json
{"timestamp": "...", "kind": "check", "file": "src/foo.rs", "rule_id": null, "status": "pass"|"warn"|"block", "elapsed_ms": 42}
```

(Rule-level breakdowns will arrive in 0.2; at 0.1c the log is per-check.)

## Process

1. Read `.ironlint/log.jsonl`.
2. Aggregate by `status` over the last N entries (default last 1000 or all if fewer).
3. Surface concerning patterns:
   - **Block rate > 50%**: rules might be too strict or scope too broad.
   - **No blocks in N runs**: rule may be dead — never fires in practice.
   - **Slow rules** (high elapsed_ms): bump or split.

## Recommendations

For each concerning rule, propose ONE of:
- **Downgrade severity**: `error` → `warning`.
- **Tighten scope**: narrow the glob so it fires only where it should.
- **Remove**: dead rules add noise to the config.
- **Investigate**: high-block-rate rules may indicate a real codebase problem to fix in source, not in the rule.

Never apply recommendations silently. Present each one and ask the user.

## Output format

```
Reviewed N entries from .ironlint/log.jsonl (date range).

| Rule | Status counts | Recommendation |
|------|---------------|----------------|
| no-console-log | pass: 412, block: 0 | DEAD — consider removing |
| no-as-any | pass: 280, block: 89 | Active; high block rate; investigate |
| audit-tests | pass: 30, block: 1 | Healthy |
```

## Limitations at 0.1c

The telemetry log doesn't yet record per-rule pass/block (only per-check). Rule-level breakdowns will be richer in 0.2. For now, the review skill is a coarse-grained signal.
```

  if [[ "${EC}" == "2" ]]; then
    echo "PASS: dirty file blocked with exit 2"
  else
    echo "FAIL: expected exit 2, got ${EC}"
    exit 1
  fi
fi

# Test 3: SessionStart clears stale session.json.

echo '{}' | "${HOOK}" session-start
if [[ -f "${PROJECT}/.ironlint/session.json" ]]; then
  echo "FAIL: session-start should have cleared session.json"
  exit 1
fi
echo "PASS: session-start clears stale state"

# Test 4: Stop with no session.json → exit 0 (no-op).
echo '{}' | "${HOOK}" stop
echo "PASS: stop with no session.json is a no-op"

# Claude Code adapter

The Claude Code adapter ships in this repo at `adapters/claude-code/`. It exposes ironlint to Claude Code as a `/plugin install` plugin.

## What it does

| Event | Action |
|-------|--------|
| `PostToolUse` (Edit\|Write) | Record the edit in `.ironlint/session.json`, then run `ironlint check --file <path>`. Exit 2 on block; the edit is rejected and the agent retries. |
| `Stop` | Run `ironlint check --session` to evaluate session-engine rules. Cleans up `.ironlint/session.json`. |
| `SessionStart` | Clear stale `.ironlint/session.json` from a prior aborted run. |

## What it does NOT do

- No subagent. Bully used a Claude Code subagent for semantic eval; ironlint calls Anthropic directly. Per `specs/2026-05-11-ironlint-plan-and-0.1-design.md` §11.
- No `Read`/`Grep`/`Glob` proxying. The adapter only invokes the `ironlint` binary.

## Install paths

### Marketplace (preferred)

Once published: `/plugin install ironlint`.

### Local development

```bash
cd /path/to/ironlint
ln -sf "$(pwd)/adapters/claude-code" ~/.claude/plugins/data/ironlint
```

Restart Claude Code.

## Requirements

- `ironlint` binary on PATH (`cargo install ironlint` or release binary).
- `jq` on PATH.
- bash.

## Skills

- `/ironlint-init` — scaffold `.ironlint.yml`.
- `/ironlint-author` — add or modify a rule, with fixture testing.
- `/ironlint-review` — audit rule health from telemetry.

## Diagnostic

If hooks aren't firing:

1. Check `${CLAUDE_PLUGIN_ROOT}/hooks/hook.sh` is executable.
2. Check `ironlint --version` runs on PATH.
3. Check `.ironlint.yml` is present in the project root.
4. Check `.ironlint.yml` is trusted (run `ironlint trust`).
5. Trace: `bash -x adapters/claude-code/hooks/hook.sh post-tool-use < event.json`.
